## 4. Критичні знахідки (потребують дій)

### F1. ПРРО-модуль ніде не задеплоєний 🔴
Baked-образ `erpnext-huntervua:v16.26.2-2` (і **прод!**) містить стару `erpnext_ua` лише з модулем
`UA FOP`. Увесь `ua_fiscal` (PRRO Settings/Shift/Receipt/Cash Register, UA KEP Key, fiscal_client,
xml_builder, offline_fiscal) існує **тільки в host-джерелі + git** (коміти `feat(fiscal)…`).
**Дія:** перед виходом ПРРО/POS на прод — перезбирати образ (Containerfile-конвеєр) з поточним
erpnext_ua і `bench migrate`. До того часу прод ПРРО не має.

### F2. Два різні API ДПС — розвилка підтверджена емпірично 🟡
- **REST (ЄВПЕЗ), прод `https://fs.tax.gov.ua:8643/fs`** — відповідає нашим клієнтом
  (`ServerState` → HTTP 200). Обов'язковий Content-Type: `application/json; charset=UTF-8`
  **або** `application/octet-stream`. Наш `fiscal_client` коректний.
- **`https://cabinet.tax.gov.ua:9443` — це gRPC** (заголовки `grpc-status: 13`, HTTP/2), а НЕ REST.
  Це «ЕК»-варіант (див. [[prro-api-two-variants]]), несумісний з нашим REST-клієнтом.
- **Наслідок:** тестування REST іде через **той самий REST-сервер із документами `<TESTING>true</TESTING>`**
  (тестові чеки нефіскальні), а не через cabinet:9443.

### F3. Дефолт `fiscal_server_test_url` вказує на gRPC 🟡
У `prro_settings.json` default `fiscal_server_test_url = https://cabinet.tax.gov.ua:9443/fs` — це
gRPC-ендпоінт (F2), REST-клієнт отримає 415. **Дія:** виправити дефолт (тест REST — через
fs:8643/fs з прапорцем TESTING) або задокументувати, що cabinet — інший протокол.

### F4. CAdES-T (TSP-мітка часу) НЕ реалізовано 🔴 — головний ризик R1
`signer.js` формує **CAdES-BES** (`op:"sign"`, `tax:false`, без TSP). За протоколом **онлайн**-чеки
вимагають CAdES-T із signature-time-stamp (TSP), офлайн-чеки — без мітки.
**Дія:** додати TSP у signer (jkurwa `op.tsp` з TSP-URL ДПС/АЦСК) до реалізації онлайн-фіскалізації.
Офлайн-режим можна тестувати вже зараз. Це головна відкрита технічна робота перед етапом 3.

## 5. Відкрита залежність (від користувача)

Повний signed-e2e (`Objects` → відкриття зміни → тестовий чек → Z-звіт) вимагає **реального тестового
ключа платника** з переліку tax.gov.ua/elektronna-zvitnist/perelik/ (тестові платники + КЕП), який
ще не завантажено в `UA KEP Key`. Це наступний крок після отримання ключа.
